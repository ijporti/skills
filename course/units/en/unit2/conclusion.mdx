## Key Takeaways

**Skill quality is not just about the body.** A skill can be perfectly well written and still feel broken if its description does not match the language users actually use.

**Project-local skills are powerful because they change the workspace, not just one turn.** The `no-fluff` example worked because you encoded a preference into the project instead of repeating yourself in every prompt.

**The core loop is understandable.** Load, select, compose is not the whole story, but it is enough to debug most activation problems intelligently.

**Tooling matters once the basics work.** `skill-creator` is useful after you already have a good skill body and want to improve how reliably it triggers.
