# Skills in Claude Code

Unit 1 showed you how to install a skill. This section is about what Claude Code does with it after that.

## Where Skills Live

Claude Code can discover the same skill at a few different scopes.

### Personal

```bash
mkdir -p ~/.claude/skills/my-skill
# put SKILL.md in this directory
```

Use this when you want the skill in every project you open.

### Project-local

```bash
mkdir -p .claude/skills/my-skill
```

Use this when the skill belongs to one repository or one team workflow. These files can live in Git alongside the code they shape.

### Plugin

```bash
/plugin marketplace add huggingface/skills
/plugin install gradio@huggingface/skills
```

Use this when someone else maintains the skill and ships it as part of a plugin. Plugin skills are namespaced as `/plugin-name:skill-name`. For this repository, the plugin name is `huggingface-skills`, so a plugin-installed skill gets a namespaced command instead of a bare `/skill-name`.

## How Activation Works

For automatic activation, the most important line in a skill is the `description` field in its frontmatter. Claude uses that text to decide whether a skill is relevant enough to load.

That makes the difference between these two versions meaningful:

```yaml
# Won't activate — too vague
description: A helper for datasets

# Better — says when to use it
description: Create and publish datasets to the HF Hub. Use when the user mentions
  publishing a dataset, uploading training data, sharing structured Q&A pairs,
  or writing a dataset card.
```

The description is not the only frontmatter field Claude understands, but it is the main auto-activation signal. Other fields can narrow or change behavior:

- `disable-model-invocation: true` stops automatic activation and keeps the skill manual-only
- `user-invocable: false` hides it from the slash-command menu
- `paths:` limits activation to matching files or folders

That means a good description should do two jobs at once:

1. Say what the skill does
2. Say when to use it in the language a user would actually type

## Slash Commands vs Auto-Activation

Skills can enter the conversation in two ways:

- **Automatic:** Claude matches your request to the skill description and loads it
- **Explicit:** You type `/skill-name` or `/plugin-name:skill-name` and force that skill into the turn

Automatic activation is what you want for background knowledge and recurring patterns. Explicit invocation is the fallback when you want a specific workflow right now.

## Checking What Is Available

Two commands are useful here:

- `/skills` shows the skills Claude Code can currently see
- `/reload-plugins` applies plugin installs, enables, or disables without restarting Claude Code

For file-based project skills, the simplest approach is still the simplest one: save the file, then continue working or start a fresh session if you want to remove any doubt about what Claude has loaded.

## Skills vs MCP

Skills are instructions. MCP servers are tools. A skill teaches Claude how to do something with the tools it already has; an MCP server gives Claude a new capability, like talking to Slack or querying a database. You will often use them together, but they solve different problems.
